5. FAILURE MODES DOCUMENTED
─────────────────────────────────────────────────────────────
These are real production failures encountered and fixed during build:

FAILURE 1 — AISPL Payment Restriction (Anthropic Claude unavailable)
  Problem:  AWS accounts registered under AISPL (Amazon Internet Services
            Pvt. Ltd., India) cannot use Anthropic Claude models on Bedrock
            due to regional payment restrictions. All Claude InvokeModel
            calls returned AccessDeniedException.
  Fix:      Switched entire reasoning and validation pipeline to Amazon
            Nova Pro (us.amazon.nova-pro-v1:0), a first-party AWS model
            available across all account types. Updated request format:
            inferenceConfig.maxTokens (not max_tokens), system as list of
            dicts, content as list of dicts with text key, response at
            raw_body['output']['message']['content'][0]['text'].
            No anthropic_version field required.
  Files:    reasoning/agent.py, evidence/threshold.py, api/routes/chat.py,
            infra/cloudformation.yaml (BedrockLLMModel parameter)

FAILURE 2 — VPC Connectivity Blocking Bedrock
  Problem:  The EvidenceLambda was initially deployed with VpcConfig
            (SecurityGroupIds + SubnetIds) to match other Lambdas. Lambda
            functions inside a VPC route outbound traffic through VPC
            endpoints or NAT. Without a Bedrock VPC endpoint configured,
            the Lambda could not reach the Bedrock API, causing all
            embedding calls to time out silently.
  Fix:      Removed VpcConfig from EvidenceLambda only. RDS is set to
            PubliclyAccessible: true for MVP, so the Lambda connects
            to RDS directly over the public endpoint. Ingestion and
            Governance Lambdas retain VpcConfig as they need DB but not
            Bedrock.
  Evidence: CloudFormation comment on EvidenceLambda reads:
            "No VpcConfig — Lambda runs outside VPC so it can reach S3,
            Bedrock, and Secrets Manager directly."
  File:     infra/cloudformation.yaml (EvidenceLambda resource)

FAILURE 3 — Lambda Handler Path Mismatch
  Problem:  The Evidence Lambda handler was initially configured as
            evidence.lambda_handler.handler but the actual function lived
            at evidence.embedding_pipeline.handler. Lambda returned
            "Unable to import module" error on every invocation.
  Fix:      Corrected Handler field in cloudformation.yaml to
            evidence.embedding_pipeline.handler.
  File:     infra/cloudformation.yaml (EvidenceLambda.Handler)

FAILURE 4 — python-multipart Missing from FastAPI
  Problem:  The FastAPI backend's CSV upload endpoint returned HTTP 422
            on all file upload requests. FastAPI requires the
            python-multipart package for multipart/form-data parsing,
            but it was absent from api/requirements.txt.
  Fix:      Added python-multipart to requirements.txt. The dependency
            is silent at import time and only manifests as a runtime
            failure when a multipart body is received.
  File:     api/requirements.txt


6. ARCHITECTURE DIFFERENTIATORS vs COMPETITORS
─────────────────────────────────────────────────────────────
What Veloquity does that Zendesk/Qualtrics/UserVoice do NOT:

A. SOURCE-TO-DECISION TRACEABILITY (evidence_item_map table)
   Schema (db/migrations/008_create_evidence_item_map.sql):
     evidence_item_map (
       evidence_id    UUID REFERENCES evidence(id),
       dedup_hash     TEXT REFERENCES dedup_index(hash),
       s3_key         TEXT,   -- exact S3 object path of the raw item
       source         TEXT,   -- 'app_store' | 'zendesk' | etc.
       item_id        TEXT,   -- original feedback item ID
       item_timestamp TIMESTAMPTZ,
       CONSTRAINT uq_evidence_item UNIQUE (evidence_id, dedup_hash)
     )
   Every recommendation traces back to specific raw feedback items via
   SHA-256 dedup_hash. PM can inspect the exact source text, source
   system, and S3 location for any cluster member. Competitors produce
   dashboards with no item-level audit trail.

B. MATHEMATICAL CONFIDENCE SCORING (evidence/confidence.py)
   Exact formula from source code:
     For each member vector v_i in cluster:
       distance_i = 1 - cosine_similarity(v_i, centroid)
     variance   = mean(distance_i for all members)
     confidence = clamp(1.0 - variance × 2.0, 0.0, 1.0)

   Three-band routing (evidence/threshold.py, env-var configurable):
     score <  CONFIDENCE_AUTO_REJECT (default 0.4): auto-reject, no LLM
     score <  CONFIDENCE_AUTO_ACCEPT (default 0.6): ambiguous → Nova Pro LLM
     score >= CONFIDENCE_AUTO_ACCEPT (default 0.6): auto-accept, no LLM

   Competitors use keyword frequency or manual tagging. Veloquity uses
   embedding-space centroid variance — tighter semantic clusters earn
   higher scores regardless of surface vocabulary.

C. REACT-STYLE REASONING LOOP (reasoning/agent.py)
   The reasoning agent is a 4-step ReAct pipeline, NOT a single prompt:
     Step 1 — Retrieve:  fetch_active_evidence(conn)
                         Reads evidence table, computes recency score
                         (days since last_validated_at).
     Step 2 — Score:     compute_priority_scores(evidence_list)
                         Formula: confidence×0.35 + users×0.25
                                  + corroboration×0.20 + recency×0.20
                         Source corroboration bonus: +0.1 if >1 source.
                         User count normalized against 50-user baseline.
     Step 3 — Build:     build_prompt(scored_evidence)
                         Constructs structured prompt with ranked clusters,
                         source lineage percentages, and JSON schema for
                         structured output.
     Step 4 — Generate:  Nova Pro invocation (maxTokens: 2000, temp: 0.2)
                         Parses structured JSON recommendation output.
                         Writes results to reasoning_runs table + S3.

   Qualtrics and Zendesk produce static rollup charts. Veloquity runs
   a prioritization algorithm that cross-correlates confidence, reach,
   source diversity, and signal recency in one pass.

D. DOMAIN-AGNOSTIC ARCHITECTURE
   The pipeline contains zero hardcoded product or domain vocabulary.
   The same Lambda code processed app product complaints (App Store +
   Zendesk) and hospital patient surveys (Patient Portal + Survey
   Tickets) without code changes. Domain is determined by the
   uploaded CSV files, not the codebase.
   Evidence in the frontend: two fully separate dataset modes
   (getActiveDataset() → 'app_product' | 'hospital_survey') with
   distinct cluster themes, different source labels, and independent
   reasoning outputs — all from the same pipeline code.

E. APPEND-ONLY GOVERNANCE AUDIT LOG (governance/governance_lambda.py)
   Schema (db/migrations/006_create_governance_log.sql):
     governance_log (
       event_type   TEXT   -- 'stale_flagged' | 'signal_promoted' | 'cache_rate_alert'
       target_id    UUID   -- nullable; references affected cluster
       details      JSONB  -- structured event payload
       actioned_at  TIMESTAMPTZ DEFAULT NOW()
     )
   Rule: NEVER DELETE or UPDATE rows in this table (CLAUDE.md Rule 6).
   Every governance action — stale signal detection, staging promotion,
   cost alerts — writes a permanent, immutable audit record. The Chat
   assistant surfaces the last 5 governance events in its context window.
   Competitors have no audit trail for how recommendations changed over time.

F. EMBEDDING CACHE WITH COST CONTROL
   Schema (db/migrations/004_create_embedding_cache.sql):
     embedding_cache (dedup_hash TEXT PK, embedding VECTOR(1024), ...)
   Bedrock is only called for new items. Re-runs on unchanged corpus
   cost near $0 for the embedding phase. Governance agent monitors
   the cache hit rate and alerts (→ governance_log) when it drops
   below CACHE_HIT_RATE_ALERT (default 40%). This is an operational
   cost-control mechanism with no equivalent in commercial survey tools.
